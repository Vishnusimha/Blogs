Q1 — Reverse an Array (In-Place)

Given: [1,2,3,4,5]

Hint 1 Use two pointers: start and end.
Answer (Thinking Steps) Swap arr[start] and arr[end] → move start++ and end-- until they meet.
Answer (Code) 
public static int[] reverse(int[] arr) {

    int start = 0;
    int end = arr.length - 1;

    while (start < end) {

        int temp = arr[start];
        arr[start] = arr[end];
        arr[end] = temp;

        start++;
        end--;
    }

    return arr;
}
Time Complexity: O(n)
Space Complexity: O(1) (In-place)

Q2 — Second Largest Element

Find second largest without sorting.

Hint 1 Track both largest and secondLargest in one loop.
Answer (Thinking Steps) If num > largest → update both largest and secondLargest. Else if num > secondLargest and num != largest → update secondLargest.
Answer (Code) 
public static int secondLargest(int[] arr) {

    int largest = Integer.MIN_VALUE;
    int secondLargest = Integer.MIN_VALUE;

    for (int num : arr) {

        if (num > largest) {
            secondLargest = largest;
            largest = num;
        }
       else if (num > secondLargest && num < largest){
            secondLargest = num;
        }
    }

    return secondLargest;
}
Time Complexity: O(n)
Space Complexity: O(1)

Q3 — Remove Duplicates (Sorted Array)

Return new length after removing duplicates.

Hint 1 Use two pointers: slow (unique position) and fast (traversal).
Hint 2 — Key Observation Array is already sorted. So duplicates will always be adjacent.
Step-by-Step Thinking (Brain Flow) Step 1: Keep slow pointer at index 0 (first unique element)
Step 2: Traverse array using fast pointer from index 1
Step 3: If arr[fast] != arr[slow], it means a new unique element
Step 4: Move slow++ and copy arr[fast] to arr[slow]
Step 5: Final new length = slow + 1
⚠️ Mistakes You Might Do (Based on Your Past Solving Style) 1. Using extra array unnecessarily (not needed, can be in-place)
2. Resetting counters like window/subArrLen (not required here)
3. Treating it like HashSet problem (overkill since array is sorted)
4. Forgetting that sorted array guarantees adjacent duplicates
5. Returning array instead of new length (problem specifically asks length)
Answer (Code) 
package ajavapractice.arrays.day1;

import java.util.Arrays;

public class RemoveDuplicatesInASortedArray {

    public static void ajavapractice.leetcode.main(String[] args) throws java.lang.Exception {

        int[] arr = {0, 0, 1, 1, 1, 2, 2, 3, 3, 4};
        System.out.println("Array length is " + arr.length);
        System.out.println("New length after removing duplicates: " + removeDuplicatesInSortedArray(arr));
    }

    public static int removeDuplicatesInSortedArray(int[] arr) {

        // Edge case handling
        if (arr == null || arr.length == 0) return 0;

        // writePointer will always point to the last unique element index
        int writePointer = 0;

        // We start from index 1 because the first element is always unique in a sorted array
        for (int i = 1; i < arr.length; i++) {

            // If current element is different from last unique element,
            // then we found a new unique value
            if (arr[i] != arr[writePointer]) {

                // Move writePointer to next position
                writePointer++;

                // Copy the new unique element to the writePointer index
                arr[writePointer] = arr[i];
            }
            // If equal, it's a duplicate → do nothing (skip)
        }

        // Debug + Learning Prints (Important for understanding in-place logic)
        System.out.println("Last unique element is: " + arr[writePointer]);
        System.out.println("Position of last unique element (writePointer): " + writePointer);
        System.out.println("---------------------- VERY IMPORTANT ------------------------");

        System.out.println("Full array after processing (duplicates still exist after unique zone):");
        System.out.println(Arrays.toString(arr));

        System.out.println("Only unique portion of array (0 to writePointer):");
        System.out.println(Arrays.toString(Arrays.copyOf(arr, writePointer + 1)));

        // VERY IMPORTANT:
        // writePointer is the index of last unique element
        // So number of unique elements = writePointer + 1
        return writePointer + 1;
    }
}
Time Complexity: O(n)
Space Complexity: O(1) (In-place)

Q4 — Rotate Array by 1 (Right)

Example: [1,2,3,4] → [4,1,2,3]

Hint 1 Store last element, then shift elements from right to left.
Step-by-Step Thinking (Your Correct Logic) Step 1: Store last element in temp
Step 2: Traverse array in reverse (n-1 to 1)
Step 3: Shift arr[i] = arr[i-1]
Step 4: Place temp at index 0
⚠️ Mistakes to Avoid (Based on Your Pattern) 1. Traversing forward while shifting (causes overwriting)
2. Forgetting to store last element before shift
3. Using extra array unnecessarily (not needed)
4. Mixing this with reverse-array logic (different problem)
Answer (Code) 
public static void rotateRightByOne(int[] arr) {

    if (arr == null || arr.length <= 1) {
        return;
    }

    // Step 1: Store last element
    int temp = arr[arr.length - 1];

    // Step 2: Shift elements to the right (reverse traversal)
    for (int i = arr.length - 1; i > 0; i--) {
        arr[i] = arr[i - 1];
    }

    // Step 3: Place last element at index 0
    arr[0] = temp;
}
Time Complexity: O(n)
Space Complexity: O(1) (In-place)

Q5 — Find Missing Number (0 to n)

Example: [3,0,1]

Hint 1 Use sum formula OR XOR.
Step-by-Step Thinking (Formula Approach) Step 1: If array has n elements, numbers range is 0 to n
Step 2: Expected sum = n * (n + 1) / 2
Step 3: Calculate actual sum of array
Step 4: Missing number = expectedSum - actualSum
Answer (Code — Using Formula) 
public static int missingNumberUsingFormula(int[] arr) {

    int n = arr.length;

    // Expected sum of numbers from 0 to n
    int expectedSum = n * (n + 1) / 2;

    // Actual sum of array elements
    int actualSum = 0;
    for (int num : arr) {
        actualSum += num;
    }

    // Missing number
    return expectedSum - actualSum;
}
Time Complexity: O(n)
Space Complexity: O(1)
Step-by-Step Thinking (XOR Approach) Step 1: XOR all numbers from 0 to n
Step 2: XOR all elements in the array
Step 3: Same numbers cancel out (a ^ a = 0)
Step 4: Remaining value is the missing number
Answer (Code — Using XOR) 
public static int missingNumberUsingXOR(int[] arr) {

    int n = arr.length;

    int xor = 0;

    // XOR all indices (0 to n)
    for (int i = 0; i <= n; i++) {
        xor ^= i;
    }

    // XOR all array elements
    for (int num : arr) {
        xor ^= num;
    }

    return xor;
}
Time Complexity: O(n)
Space Complexity: O(1)
⚠️ Mistakes to Avoid (Based on Your Solving Pattern) 1. Using sorting first (unnecessary O(n log n))
2. Forgetting that range is 0 to n (not 0 to n-1)
3. Using extra HashSet (overkill for this problem)
4. Integer overflow risk if n is very large (formula method)
5. Confusing length vs max value (n = arr.length, not max element)

Q6 — Maximum Difference (j > i)

Find maximum value of arr[j] - arr[i] such that j > i.

Hint 1 Track minimum element seen so far while traversing.
Step-by-Step Thinking (Brain Flow) Step 1: Initialize minSoFar = arr[0]
Step 2: Initialize maxDiff = Integer.MIN_VALUE
Step 3: Traverse from index 1 to end
Step 4: At each element, calculate diff = arr[i] - minSoFar
Step 5: Update maxDiff if diff is larger
Step 6: Update minSoFar = min(minSoFar, arr[i])
⚠️ Mistakes to Avoid (Based on Your Past Pattern) 1. Using nested loops (O(n²)) — unnecessary
2. Trying sliding window logic (not required here)
3. Updating minSoFar after calculating diff incorrectly
4. Forgetting that j must be greater than i
5. Confusing this with maximum subarray (different problem)
Answer (Code) 
public static int maximumDifference(int[] arr) {

    if (arr == null || arr.length < 2) {
        return 0;
    }

    int minSoFar = arr[0];
    int maxDiff = Integer.MIN_VALUE;

    for (int i = 1; i < arr.length; i++) {

        // Calculate difference with smallest value seen so far
        int diff = arr[i] - minSoFar;

        maxDiff = Math.max(maxDiff, diff);

        // Update minimum value for future comparisons
        minSoFar = Math.min(minSoFar, arr[i]);
    }

    return maxDiff;
}
Time Complexity: O(n)
Space Complexity: O(1)

Q8 — Best Time to Buy and Sell Stock

Max profit with one transaction.

Hint 1 Same logic as maximum difference (j > i).
Step-by-Step Thinking (Brain Flow — How You Should Think) Step 1: This is NOT a sliding window problem
Step 2: You can only buy first, then sell later (order matters)
Step 3: Track the minimum price seen so far (best buying price)
Step 4: At each day, calculate profit = currentPrice - minPriceSoFar
Step 5: Update maxProfit if this profit is higher
Step 6: Update minPriceSoFar for future days
⚠️ Mistakes You Are Likely to Make (Based on Your Previous Code) 1. Using wrong condition to update minimum (like if(min < arr[i])) ❌
2. Updating min AFTER profit incorrectly
3. Starting with Integer.MAX_VALUE and calculating profit at i=0
4. Confusing this with Kadane (subarray problem) ❌
5. Trying nested loops (O(n²)) instead of single pass
Deep Insight (Pattern Recognition) This problem is mathematically identical to:
Maximum Difference (j > i)

Mapping:
minPriceSoFar = buy price
current price = sell price
maxProfit = best profit
Answer (Code) 
public static int maxProfit(int[] prices) {

    if (prices == null || prices.length < 2) {
        return 0;
    }

    int minPriceSoFar = prices[0];
    int maxProfit = 0;

    for (int i = 1; i < prices.length; i++) {

        // Calculate profit if sold today
        int profit = prices[i] - minPriceSoFar;

        // Update maximum profit
        maxProfit = Math.max(maxProfit, profit);

        // Update minimum price for future days (buy cheaper)
        minPriceSoFar = Math.min(minPriceSoFar, prices[i]);
    }

    return maxProfit;
}
Time Complexity: O(n)
Space Complexity: O(1)
Dry Run (So It Sticks Permanently) Input: [7,1,5,3,6,4]

Day 0: min = 7
Day 1: min = 1
Day 2: profit = 5 - 1 = 4 → maxProfit = 4
Day 4: profit = 6 - 1 = 5 → maxProfit = 5 (BEST)

Q9 — Find Majority Element

Element appears > n/2 times.

Hint 1 Do NOT use XOR. Majority is not a pair-cancellation problem like Single Number.
Hint 2 (Key Observation) If an element appears more than n/2 times, it will dominate the array and cannot be fully cancelled by other elements.
Step-by-Step Thinking (How Your Brain Should Work) Step 1: Full frequency storage (HashMap) is not necessary
Step 2: We only need to track a potential dominant candidate
Step 3: Traverse array once
Step 4: If count becomes 0 → choose new candidate
Step 5: If element equals candidate → increment count
Step 6: Else → decrement count (cancel out different elements)
⚠️ Mistakes You Are Likely to Make (Based on Your Previous Attempts) 1. Using XOR to find majority ❌ (works only for pair problems)
2. Using HashMap when O(1) space solution exists
3. Confusing this with frequency counting problems
4. Forgetting that > n/2 guarantees dominance
5. Returning candidate without verification when majority is NOT guaranteed
Answer (Code — Normal intuitive approach) 
    public static int findMajorityElement(int[] arr) {

        if (arr == null || arr.length < 2) {
            return -1;
        }

        // int majorityElement = Integer.MIN_VALUE;
        int threshold = arr.length / 2;

        HashMap map = new HashMap<>();


        for (int i = 0; i < arr.length; i++) {
            map.put(arr[i], map.getOrDefault(arr[i], 0) + 1);
            if (map.get(arr[i]) > threshold) {
                return arr[i];
            }
        }

        return -1;
    }
Time Complexity: O(n)
Space Complexity: O(n) (due to HashMap)
Answer (Code — Boyer-Moore Voting Algorithm) 
public static int findMajorityElement(int[] arr) {

    if (arr == null || arr.length == 0) {
        return -1;
    }

    int candidate = 0;
    int count = 0;

    // Phase 1: Find potential candidate
    for (int num : arr) {

        if (count == 0) {
            candidate = num;
        }

        if (num == candidate) {
            count++;
        } else {
            count--;
        }
    }

    return candidate;
}
Time Complexity: O(n)
Space Complexity: O(1)
Answer (Code — With Verification Pass, Safer Version) 
public static int findMajorityElement(int[] arr) {

    if (arr == null || arr.length == 0) {
        return -1;
    }

    int candidate = 0;
    int count = 0;

    // Phase 1: Find candidate
    for (int num : arr) {
        if (count == 0) {
            candidate = num;
        }

        if (num == candidate) {
            count++;
        } else {
            count--;
        }
    }

    // Phase 2: Verify candidate (important if majority not guaranteed)
    int freq = 0;
    for (int num : arr) {
        if (num == candidate) {
            freq++;
        }
    }

    if (freq > arr.length / 2) {
        return candidate;
    }

    return -1;
}
Time Complexity: O(n) + O(n) = O(n)
Space Complexity: O(1)
Deep Pattern Insight (Very Important for Interviews) This problem belongs to the Dominance / Voting Pattern.
If one element appears more than n/2 times, all other elements combined cannot cancel it out during pairwise elimination, so it survives as the final candidate.

Q10 — Product of Array Except Self 🟡 Medium (30–45 mins each)

No division allowed.

Hint 1 Think mathematically: result[i] = product of left side × product of right side.
Hint 2 Can you reuse computations instead of recalculating left and right for every index? (Avoid O(n²) nested loops.)
Hint 3 First pass: store prefix product in result array. Second pass (reverse): multiply running suffix product into result.
Thinking Steps (Pseudo Logic) 1. Understand result[i] = product of all elements except arr[i]
2. Break it into:
    Left product (0 → i-1)
    Right product (i+1 → n-1)
3. Use result[] as prefix storage in first pass
4. Traverse from right and maintain suffix product
5. Multiply suffix into existing result[i]
6. Do NOT include arr[i] in its own product
Answer (Optimal Approach) Use two passes:
Pass 1 → store prefix product in result array.
Pass 2 → traverse from right and multiply running suffix product.
No division, O(n) time, O(1) extra space.
Answer (Code — Optimal) 

public static int[] productExceptSelf(int[] arr) {
    int n = arr.length;
    int[] result = new int[n];

    // Step 1: Prefix pass (left → right)
    result[0] = 1; // no elements on the left
    for (int i = 1; i < n; i++) {
        result[i] = result[i - 1] * arr[i - 1];
    }

    // Step 2: Suffix pass (right → left)
    int suffix = 1;
    for (int i = n - 1; i >= 0; i--) {
        result[i] *= suffix;   // use suffix first (right side product)
        suffix *= arr[i];      // then update suffix
    }

    return result;
}
Time & Space Complexity Time Complexity: O(n) (Two linear passes)
Space Complexity: O(1) extra (Output array not counted)
Common Mistakes (Very Important — Based on Real Solving Errors) • Using nested loops to calculate suffix for every index (O(n²))
• Initializing product as 0 instead of 1 (kills multiplication)
• Including current element in suffix calculation
• Updating suffix before using it on result[i]
• Using arr[i+1] or arr[i-1] incorrectly (off-by-one errors)
• Resetting suffix inside loop instead of maintaining running suffix
• Manually fixing index 0 instead of designing correct two-pass logic
• Overwriting result instead of multiplying prefix × suffix
Pattern Recognition (Interview Note) Pattern: Two-Pass Accumulation (Prefix + Suffix)
Not: Sliding Window ❌
Not: HashMap ❌
Not: Two Pointer Sync ❌
Category: Array + Prefix Products + Space Optimization
